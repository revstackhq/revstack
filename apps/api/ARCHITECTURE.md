Revstack API: Architecture Guide (Agent-Native)

The architecture of the `@revstackhq/api` project follows a **Feature-Sliced Design (Vertical Architecture)** pattern, heavily influenced by **Domain-Driven Design (DDD)**, **Hexagonal Architecture (Ports and Adapters)**, and **CQRS (Command Query Responsibility Segregation)**.

Each business domain (e.g., `customers`, `plans`, `invoices`) operates independently, with self-contained layers. As an AI Agent, **you must strictly respect these structural and naming conventions when generating or refactoring code**.

## 📂 Base Directory Structure

```text
src/
├── index.ts        # Entry point. Configures Hono, OpenAPI, and mounts routes.
├── container.ts    # DI Container. Instantiates repositories and orchestrates use cases.
├── common/         # Shared Kernel. Generic code (base events, base repositories, errors).
└── modules/        # Feature Slices. Independent business domains.
1. The common/ Directory (Shared Kernel)
Contains the fundamental pieces that are agnostic to individual modules.

application/ports/: Base interfaces for cross-cutting services (e.g., EventBus.ts, CacheService.ts).

domain/: Abstract base classes such as Entity<T> and DomainEvent.

errors/: Common errors mapped to HTTP codes.

infrastructure/adapters/: Generic implementations. Highlights BasePostgresRepository.ts, which exposes CRUD behavior isolated by environmentId (multi-tenant).

middlewares/: Global Hono middlewares (e.g., API Key validation, environmentId injection).

2. The modules/ Directory (Feature Slices)
Each folder within modules/ represents a business context (e.g., customers) and is divided into 3 fundamental layers.

A. Domain Layer (domain/)
The core of the hexagon. Contains pure logic. Importing HTTP frameworks or Database libraries here is strictly forbidden.

[Feature]Entity.ts: Rich models that maintain business invariants.

CRITICAL RULE 1: All entity properties MUST be in strictly camelCase (e.g., environmentId, not environment_id).

CRITICAL RULE 2: The id property in the Entity Props MUST be optional (id?: string). Do not generate IDs manually (e.g., no crypto.randomUUID()); let the ORM (Drizzle) handle ID generation at the database level.

[Feature]Errors.ts: Native domain errors.

events/[Feature]Events.ts: Consolidated file with all domain events. They must extend DomainEvent.

Example of Entity (CustomerEntity.ts):

TypeScript
import { Entity } from "@/common/domain/Entity";

export interface CustomerProps {
  id?: string; // Optional, generated by the database
  environmentId: string; // strictly camelCase
  email: string;
}

export class CustomerEntity extends Entity<CustomerProps> {
  // Methods mutate state and generate events...
}
B. Application Layer (application/)
Defines use cases implementing CQRS. It is organized inside a use-cases/ folder.

ports/: Infrastructure contracts (e.g., CustomerRepository.ts).

use-cases/[Action]/: Dedicated folder for each operation (Command or Query). It must always contain two files: the Schema and the Handler.

Rules for Schemas ([Action].schema.ts):

They must define both the Input DTO (Command/Query) and the Output DTO (Response) using Zod.

CRITICAL RULE 3: All Zod schema properties for queries, commands, and DTOs MUST strictly use snake_case to map cleanly to HTTP/JSON requests.

TypeScript
// use-cases/CreateUser/CreateUser.schema.ts
import { z } from 'zod';

// 1. Input DTO (Command) - strictly snake_case
export const CreateUserCommandSchema = z.object({
  email: z.string().email(),
  environment_id: z.string().uuid()
});
export type CreateUserCommand = z.infer<typeof CreateUserCommandSchema>;

// 2. Output DTO (Response) - strictly snake_case
export const CreateUserResponseSchema = z.object({
  id: z.string().uuid(),
  email: z.string().email(),
  created_at: z.date()
});
export type CreateUserResponse = z.infer<typeof CreateUserResponseSchema>;
Rules for Handlers ([Action].handler.ts):

The main method must be named execute.

The function contract must be strongly typed using the inferred types from the Zod schema.

The handler is responsible for translating the snake_case DTO into the camelCase entity properties when instantiating the domain.

C. Infrastructure Layer (infrastructure/)
Implements the ports (DB connections) and Web adapters (Hono Routes).

adapters/Postgres[Feature]Repository.ts: Repository implementation using Drizzle. It must extend BasePostgresRepository.

http/: Web Controllers layer.

CRITICAL RULE 4: Routes and Controllers MUST be separated. [feature].routes.ts contains only the OpenAPI definitions, and [feature].controller.ts contains the execution logic using c.req.valid().

1. [feature].routes.ts (OpenAPI Declaration):

TypeScript
import { createRoute, z } from "@hono/zod-openapi";
import { updateRefundSchema } from "../../application/use-cases/UpdateRefund/UpdateRefund.schema";

export const updateRefundRoute = createRoute({
  method: "patch",
  path: "/{id}",
  tags: ["Refunds"],
  summary: "Update a refund",
  description: "Updates metadata or status of an existing refund.",
  request: {
    params: z.object({ id: z.string().openapi({ example: "ref_abc123" }) }),
    body: { content: { "application/json": { schema: updateRefundSchema } } },
  },
  responses: {
    200: { description: "Refund updated", content: { "application/json": { schema: z.any() } } }
  },
});
2. [feature].controller.ts (Endpoint Logic):

TypeScript
import { OpenAPIHono } from "@hono/zod-openapi";
import { apiKeyMiddleware } from "@/common/middlewares/api-key";
import type { AppEnv } from "@/container";
import { updateRefundRoute } from "./refunds.routes";

export const refundsController = new OpenAPIHono<AppEnv>();

refundsController.use("*", apiKeyMiddleware);

refundsController.openapi(updateRefundRoute, async (c) => {
  const handler = c.get("refunds").update;

  // Use c.req.valid to extract strictly typed and validated Zod inputs from the route
  const { id } = c.req.valid("param");
  const dto = c.req.valid("json");

  const result = await handler.execute({ id, ...dto });
  return c.json(result, 200);
});
3. index.ts (Router Exporter):

TypeScript
import { refundsController } from "./refunds.controller";
export const refundsRoutes = refundsController;
🔄 Request Lifecycle
Entry: The main index.ts receives the request and routes it to the corresponding [feature].controller.ts.

Web Layer: The middleware injects the environmentId. The controller extracts and validates the parameters and request body using c.req.valid(), mapped against the Input Schema ([Action].schema.ts), and passes them to the Handler.

Use Case (Application): The [Action].handler.ts executes its execute() method. It coordinates dependency loading.

Domain: Entities ([Feature]Entity) are instantiated by translating snake_case to camelCase. Their methods are executed to validate business rules and generate DomainEvents.

Persistence: The Handler injects the repository (Postgres[Feature]Repository.ts), which translates the request to SQL using Drizzle and saves it in the database while respecting environment isolation.

Output & Effects: The Handler filters and formats the output data (back to snake_case) using the Response Schema (ResponseDTO), emits the stored events to the EventBus, and returns the result to the controller to send the HTTP response.
```

## 3. General Coding Guidelines

### A. Path Aliases

- **CRITICAL RULE 5:** Always use path aliases for imports.
- Use `@/common/...` for shared kernel and `@/modules/[feature]/...` for module-specific code.
- **Strictly avoid** deep relative paths (e.g., `../../../../common/domain/Entity`).

### B. Domain Event Payloads Casing

- **CRITICAL RULE 6:** Domain Events are public contracts. Their payloads MUST be strictly **`snake_case`**.
- The Entity is responsible for mapping its internal `camelCase` state to the `snake_case` event payload when instantiating the event.

**Correct Implementation Example:**

```typescript
// Inside an Entity method:
this.addEvent(
  new AddonEntitlementCreatedEvent({
    addon_id: this.props.addonId, // Mapping camelCase -> snake_case
    entitlement_id: this.props.entitlementId,
    environment_id: this.props.environmentId,
  }),
);
```
